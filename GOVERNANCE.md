# 工程治理与执行纪律 (Governance & Discipline)

本文件规定了 GPAO 系统在执行期间必须遵守的工程安全与发布纪律。所有 Agent 在操作本项目文件时，必须无条件遵守以下条款。

## 1. 自动继续 (Auto-proceed) 约束
- **原则**：任何需要用户确认的步骤（例如：“等待您审查”、“在您批准后”），必须在技术上**完全停止运行**。
- **强制阻断**：绝不可将平台的“自动继续执行 (Auto-proceed)”机制视为用户的显式授权同意。
- **明确授权标准**：必须要求用户输入明确的授权口令（如：“批准执行”、“同意计划”）。如果未收到包含此类意图的用户文本回复，则视为未授权，禁止进行后续写操作。

## 2. 破坏性操作与备份纪律
- **破坏性操作定义**：任何删除（Delete）、覆盖（Overwrite）、重命名（Rename）或全盘重构。
- **强制前置条件**：在执行破坏性操作前，必须建立并**验证**外部快照（ZIP）。
- **失败立即停止**：如果备份过程中发生任何错误（如路径过长、权限拒绝、无限递归），则必须立即停止并向用户报告。**失败的备份不叫备份，没有成功备份时严禁继续执行删除操作。**

## 3. 版本发布与回滚
- **双重备份机制**：
  - **灾难备份**：项目外部的 ZIP 归档，附带 SHA-256 清单。用于应对文件系统级崩溃（如递归循环损坏）。
  - **日常版本控制**：Git。
- **发布条件**：每次修改后，必须生成 diff、更新文件哈希清单，并建立新的回滚点（Git commit / tag）。
- **虚假承诺禁止**：严禁在发布说明（CHANGELOG 或报告）中，将“自动推进执行”包装或描述成“在用户批准后执行”。必须如实记录每次操作的授权状态。
