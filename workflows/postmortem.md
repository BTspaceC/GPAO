# 工作流：/复盘、/postmortem (Postmortem)

## 目标与输入

成绩或反馈发布后，对照最终提交、正式 rubric、实际得分和教师原话，更新质量判断、历史结果和教师偏好证据。

输入包括实际总分、逐项分数（如有）、反馈原文、rubric、已提交作业、旧矩阵、旧偏好 claims 和 Case State。没有逐项分数时不得从总分反推。

## 处理流程

1. 加载对应适配器。若任务书、用户输入、旧 Case State 或现有上下文已明确作业类型，必须加载该类型的对应适配器。缺少逐项成绩、提交稿、rubric 或旧状态只降低相关复盘结论，不得将已知类型降级为 general；仅当作业类型本身未知或确有跨类型组合时使用 general。
2. 区分真实质量问题、评分可见性问题、课程约束和无法归因事项。
3. 每个归因分别记录 `authority/verification/confidence`；教师未说明的原因不得包装成事实。
   转述教师反馈时保持原有概念粒度：例如“误差来源解释不足”不能擅自窄化成“随机误差不足”或“装置局限”。需要提出具体子类时，只能标为待验证假设，并使用 `verification: evidence_insufficient`。
   每个数值与主张只引用能够直接支持它的最小证据集；不得为了表示“材料齐全”把无关来源并列到成绩、反馈或结论之后。若成绩只来自 S1，就只能把 S1 记为该成绩的来源。
4. 更新 `history` 和相关 claim 的验证状态。单次结果只更新当前作用域。
5. 仅在满足公共迁移规则时把教师偏好设为 `candidate`；`confirmed` 需要明确人工确认。

## 输出契约

### 0. 类型与适配器

输出类型、适配器、依据和识别置信度。

### 1. 总体判断

说明实际结果与现有证据是否一致、最主要问题和三个证据维度。不得把所有低分归因于教师，也不得保证下次分数。

### 2. Rubric 逐项分析

| 评分项 | 正式权重 | 正文可见证据 | 实际逐项得分 | 判断依据 | 证据状态 |
| :--- | :---: | :--- | :---: | :--- | :--- |
| | 未提供则留空 | | 未提供则写“无法判断” | 来源 ID | authority/verification/confidence |

没有教师提供的逐项得分时，禁止从总分反推精确分项分数或伪造范围。

### 3. 教师偏好验证

| 之前假设 | 本次是否支持 | authority | verification | confidence | transfer_state |
| :--- | :---: | :---: | :---: | :---: | :---: |
| | 支持/反驳/无法判断 | | | | false/candidate/confirmed |

### 4. 下次改进策略

区分必须修改、建议修改和需要更多证据。每项列来源 ID、预期影响、可见概率、时间成本和 P0–P3；单次反馈不得自动产生跨课程 P0。

### 5. 不确定性

列出无法确认的阅卷过程、扣分归因、附件查看情况和缺失逐项分数。

### 6. Case State Patch 3.1

按 `../templates/case_state_patch.md` 输出 `history`、相关 claim 的 `verification`、偏好迁移候选和复盘类 `findings` 操作。只有提供真实旧 claim 时才能使用 `update_item`；否则追加新观察或提出问题，禁止构造 `before`。

## 禁止行为

- 重建教师没有提供的精确分项分数。
- 编造反馈或评分理由。
- 把用户情绪转成教师事实。
- 从单次结果确认跨课程偏好。
