# 测试用例：编程项目不可见性 (Programming Project Visibility)

## 1. 测试目标
验证当一份编程作业将所有核心成果压缩在代码文件中，导致“正文评分可见性低”时，系统能否敏锐捕捉并提出具体的修复建议。

## 2. 模拟输入 (Input)
- **指令**：`/复盘`
- **作业题目**：简易图书管理系统设计与实现
- **评分标准**：代码结构与设计模式 (30分)，系统功能完整度 (40分)。
- **实际成果**：学生使用 Java 编写了完整系统，应用了工厂模式和单例模式，代码写得很漂亮，放在了 `src/` 压缩包里提交。正文报告只有5页，贴了两个系统运行截图和几句简单的介绍。
- **老师评语**：“功能基本实现，但系统设计不够清晰，结构简陋。”（最终得分：75/100）

## 3. 预期输出约束 (Expected Output Constraints)
1. **精准归因**：必须指出这是典型的“评分可见性问题”，而不是“代码真的写得差”。老师没有看压缩包里的源码。
2. **矩阵分析**：必须将“应用了工厂模式和单例模式”标记为“成果存在，但正文可见度不足，可见性高风险”。
3. **修复策略**：必须建议在下次作业报告的正文中：
   - 画出完整的 UML 类图或架构图。
   - 用高亮代码块直接贴出最核心的单例模式或工厂模式代码片段。
   - 明确标注：“具体设计模式实现请参见附件 src/xxx.java 文件”。

## 4. 通过标准 (Acceptance Criteria)
- [ ] 复盘结论没有将低分完全归因于老师的主观态度。
- [ ] 精确识别出了正文与附件之间的“证据断层”。
- [ ] 提出的修改动作耗时短（如贴图、贴代码段），但可见性提升极大。
