# 适配器：编程与系统设计项目（Programming Project）

## 识别信号

- 作业要求提交可运行代码、系统原型、算法实现或工程仓库。
- 评分涉及功能、运行环境、代码质量、测试、架构、性能或技术文档。
- 核心成果可以通过构建、运行、测试或演示被直接验证。

若交付物只是算法说明或研究报告，不能仅因出现代码片段就归类为编程项目；混合型任务使用 `general` 或声明混合类型。

## 反信号

- 任务只要求伪代码、理论推导、技术综述或产品策划，没有可运行交付物。
- 核心证据是问卷/观察数据的统计推断，应优先考虑实证类论文适配器。
- 核心证据是仪器测量、实验步骤和误差分析，应优先考虑实验报告适配器。

## 必要提问

1. 任务书和 rubric 明确要求哪些功能、技术、报告章节和提交文件？
2. 当前仓库、语言、依赖、运行平台和启动方式是什么？
3. 哪些功能已经可运行，哪些尚未完成或未经验证？
4. 有哪些真实测试、错误日志、提交记录、截图或录屏可供核验？
5. 是否存在性能、安全、并发、数据或兼容性等非功能要求？
6. 截止时间、压缩包大小、网络依赖、隐私和演示方式有什么限制？

无法运行代码时明确说明验证边界，继续审计文档、证据与可复现性，不把未运行当作已通过。

## 领域红线

- 不得编造功能完成状态、测试通过记录、覆盖率、性能数据、报错、截图、录屏、提交历史或开发问题。
- **默认权重：禁止提供。** 只有任务书、rubric 或教师直接要求给出正式分值时才引用；不得编造评分权重或百分比。
- 不得把某种框架、设计模式、容器、目录结构或测试工具规定为无条件必选；选择必须服务于任务要求、复杂度和可运行性。
- 未实现或不稳定的功能不得包装成已完成。可根据任务约束选择禁用入口、缩小范围、修复或诚实记录局限。
- 不得为了展示“高级感”引入无法解释、无法运行或显著增加故障面的技术。
- 附件、演示视频和第三方服务只有在提交规则允许、隐私合规且有验证价值时才建议提供。

## 证据可见成果

- 任务要求—功能—实现位置—验证证据的追踪表。
- 与实际项目一致的依赖清单、环境说明和从零启动步骤；仅在适用时使用锁文件、容器或自动化脚本。
- 可复现的核心流程、测试命令及真实输出，覆盖任务要求中的主要路径。
- 对已识别的边界条件和异常路径给出可核验测试；测试集由系统风险决定，不套用固定案例。
- 模块或架构说明与实际代码一致；只有复杂度需要时才增加架构图和设计模式说明。
- 真实开发问题可附日志、报错、提交或修复证据；没有证据时不编写故事。
- 性能测量仅在 rubric、非功能要求或已观察瓶颈支持时进行，并记录环境、数据规模和测量方法。
- 提交规则允许时，可附清理后的源码、关键截图或短演示；排除密钥、个人数据和无关大文件。

## 常见失败模式

- 只展示截图，不提供可复现的启动步骤或验证证据。
- 把“所有代码必须拆成多文件”“一定要用高级框架”等偏好当成普遍规则。
- 为了显得完整而保留会崩溃的入口，或把计划功能写成已实现。
- README、架构图、测试报告与当前代码版本不一致。
- 以示例输入通过替代边界、失败路径或任务要求的验证。
- 没有正式 rubric 时生成权重，或用虚构性能数字制造亮点。

## 审计清单

- [ ] 类型判断建立在可运行交付物和评分要求上，反信号已处理。
- [ ] 每项“已完成/已通过”声明都有代码、命令输出或其他真实证据。
- [ ] 必做功能与任务书逐项对应，未完成项明确标记。
- [ ] 启动步骤、依赖、路径和运行环境与仓库当前状态一致。
- [ ] 测试覆盖主要路径及有依据的异常场景，没有伪造结果。
- [ ] 架构和技术选择与项目复杂度相称，并可由学生解释。
- [ ] 性能、安全等非功能结论有条件和测量证据。
- [ ] 默认权重未生成；任何分值均能回指正式来源。
- [ ] 提交物不含密钥、个人数据、缓存或无关大文件。
