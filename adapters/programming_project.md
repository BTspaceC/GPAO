# 适配器：编程与系统设计项目 (Programming Project)

## 适用范围
计算机、软件工程、信息系统等专业的课程大作业，通常要求提交可运行的代码和说明文档。

## 核心评分点转换表

| 常见评分点 | 在 GPAO 中的等效要求 | 报告/正文应体现的位置 |
| :--- | :--- | :--- |
| **功能完整性 (30%)** | 任务书要求的所有基础功能必须跑通。 | "系统功能说明"与"运行截图"小节 |
| **代码质量与规范 (20%)** | 目录结构清晰，有合理的注释，无明显硬编码。 | 根目录结构树，及核心算法的说明 |
| **架构与设计模式 (20%)** | 不能所有代码写在一个文件里。要有基础的模块划分。 | "系统架构图"或"模块设计"小节 |
| **异常处理与鲁棒性 (15%)** | 必须展示当输入错误时，系统不会崩溃而是给出友好提示。 | "异常处理与测试"小节 |
| **文档完整性 (15%)** | README 必须包含从零启动的环境配置和运行指南。 | 根目录 README.md |

## 高辨识度亮点设计池 (P2 优先级)

1. **环境一键配置**：提供 `requirements.txt`、`package.json` 或 `docker-compose.yml`，降低老师运行代码的门槛。
2. **丰富的异常测试记录**：在文档中专门开辟“边界条件测试”章节，截图展示系统应对非法输入（如除以零、文件不存在、SQL注入模拟）的表现。
3. **开发问题与解决记录**：记录开发过程中遇到的真实 bug。必须有日志、截图、提交记录或报错信息支持，不得编造。
4. **性能基准测试**：简单的耗时统计或内存占用曲线，体现对非功能性需求的关注。



**默认权重**：禁止提供。仅当任务书、评分表或老师明确要求中存在分值时填写。没有正式权重时，只能描述该项可能的重要性，不得生成具体百分比。

## 常见冲突处理

- **代码不完美 vs 强行掩饰**：如果某个功能无法在截止前完成，不要留一个会报错的按钮。应该在界面上移除该入口，并在文档的“未来改进”或“局限性”中诚实说明。
- **高级框架偏好 vs 自身能力**：老师偏好最新框架，但使用者只会基础语言。优先保证基础语言写出的系统稳定运行，文档中体现良好的设计思维（如面向对象），这比使用高级框架但跑不起来得分更高。

## 附件与提交要求

为了确保“代码可见性与可验证性”，必须提交：
1. **完整源码**：清理掉 `.git`、`node_modules` 等无关大文件。
2. **运行截图/录屏**：部分老师不愿配置环境，**必须**在文档正文中附带核心操作流程的截图，或在根目录提供一个 1-2 分钟的 MP4 演示视频。
*(正文中必须写明："完整源码和运行指南见压缩包根目录，演示视频为 demo.mp4")*
